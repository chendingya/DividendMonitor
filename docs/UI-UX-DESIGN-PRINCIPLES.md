# 收息佬 UI/UX 设计原则（贴合 Stitch + 当前实现）

## 1. 文档目的与适用范围

本文档用于统一收息佬桌面端（Electron + React）的 UI/UX 设计决策，目标是：

1. 视觉风格尽可能贴合 `stitch_yieldmaster_frontend_ui`
2. 与当前 `src/renderer` 已落地结构保持一致，避免“只好看不可实现”
3. 为后续页面迭代提供统一的设计和验收标准

适用范围：

1. 左侧导航、顶部栏、页面主画布
2. 指标卡、数据表、筛选条、图表、解释面板
3. 状态反馈（loading / empty / error / no-data）

---

## 2. 设计北极星

关键词：清晰、轻盈、专业、可解释。

1. 清晰：先让用户看懂“收益结论”，再看“计算过程”
2. 轻盈：大量留白、柔和层级、少硬边线
3. 专业：数据展示可信、结构稳定、交互一致
4. 可解释：关键指标必须能追溯口径和来源

一句话原则：不是做“信息堆叠面板”，而是做“可连续决策的收益工作台”。

---

## 3. 视觉语言原则

## 3.1 色彩与层级

1. 主品牌色固定为蓝色系：`--primary` / `--primary-dim`
2. 背景使用浅灰白层级：`--app-bg`、`--surface`、`--surface-low`
3. 语义色只用于状态表达：成功、风险、告警，不用于装饰
4. 保持“主色聚焦”：页面中最重要的行动和关键增长值才使用主色高强调

## 3.2 无硬分割线优先

1. 优先使用留白、背景层差、圆角卡片来分区
2. 表格和列表仅使用弱分隔（低对比度）
3. 禁止大面积高对比 1px 边线切割内容

## 3.3 玻璃感与柔和阴影

1. 侧边栏、顶部栏、Hero 卡片使用半透明 + blur
2. 阴影使用大半径、低不透明度，避免“厚重浮层感”
3. 浮层感服务于信息层级，不服务于炫技

## 3.4 圆角体系

1. 页面级容器：大圆角（22-30）
2. 卡片与模块：中圆角（16-22）
3. 输入与小控件：小圆角（8-12）
4. 胶囊按钮、导航激活态、标签统一使用全圆角

---

## 4. 字体与排版原则

1. 标题使用 Plus Jakarta Sans 风格（当前以同类字重和字距模拟）
2. 正文与数据说明使用 Inter / 中文系统字体回退
3. 数字优先右对齐或等宽视觉对齐（金额、比例、日期）
4. 标题字重高、字距紧；说明字重中等、字距常规
5. 文案层级固定为：页面标题 > 区块标题 > 指标值 > 标签说明 > 辅助注释

---

## 5. 页面骨架原则

## 5.1 全局骨架

1. 左侧固定导航：承载主任务入口，不放页面内临时操作
2. 顶部栏：搜索 + 通知 + 全局操作，占位稳定
3. 主画布：以卡片编排信息块，按“先总览后明细”从上到下布局

## 5.2 任务流顺序

所有核心页面遵循同一阅读顺序：

1. 先看结论（核心指标）
2. 再看趋势（图表）
3. 再看明细（表格）
4. 最后看解释（口径、公式、假设）

---

## 6. 组件设计原则

## 6.1 导航与图标

1. 一级导航必须“图标 + 文案”双编码
2. 激活态必须具备三个信号：主色底、白色文案、轻阴影
3. 图标风格统一为线性圆角体系，不混用多套风格
4. 不允许仅靠颜色区分入口，必须保留文字语义

## 6.2 指标卡（Metric Card）

1. 每张卡只表达一个主指标
2. 卡片结构固定：标签 -> 数值 -> 解释/状态
3. 主卡（primary）用于“最关键收益指标”，同屏最多 1 张
4. 指标说明用短语，不写长段落

## 6.3 表格与列表

1. 表头语义清晰，字段命名稳定
2. 金额、比例、日期对齐方式一致
3. 行 hover 提示可用，但强度克制
4. 列表默认展示“最相关、最近、最可比较”的数据
5. 长列表必须分页或分段，避免长滚动疲劳

## 6.4 筛选与标签

1. 筛选项使用胶囊按钮，激活态必须明显
2. 状态标签使用低饱和底色 + 高可读文字
3. 标签用于分类和风险提示，不替代表格字段

## 6.5 图表

1. 先服务理解，再追求装饰
2. 同图中系列不超过 2-3 条，颜色语义稳定
3. 趋势图与明细表必须互相印证，不出现口径冲突
4. 图表下方保留一句口径说明

## 6.6 解释面板

1. “计算过程”必须是人话步骤，不是公式堆叠
2. 推荐结构：前提 -> 计算 -> 结果
3. 关键输入值应可见，避免黑盒感

---

## 7. 交互与反馈原则

1. 所有可点击区域必须有 hover/active 反馈
2. 重要操作按钮位置稳定，不随内容跳动
3. 页面内跳转尽量保留上下文（股票代码、筛选条件）
4. 错误提示说人话：说明失败原因 + 可执行下一步
5. 加载态和空态优先于“白屏等待”

---

## 8. 数据可读性与可信度原则

1. 所有收益指标必须说明统计口径
2. 同一概念在不同页面命名一致（如“自然年股息率”）
3. 数据缺失要显式暴露，不做静默替换
4. 时间序列默认“最近在前”用于业务阅读，趋势图可按时间正序绘制
5. 回测和预测结果都要给出前提假设，不制造确定性错觉

---

## 9. 文案与语言风格原则

1. 中文优先，术语稳定，避免中英文来回切换
2. 操作文案用动词开头：查看、添加、导出、解释
3. 指标说明简洁具体，不写空泛营销词
4. 口径描述统一模板：按什么时间、按什么数据、如何计算

---

## 10. 响应式与桌面适配原则

1. 首要优化桌面宽屏（当前主场景）
2. 中等宽度下允许栅格降列，但不改变信息顺序
3. 小屏时优先保证可读和可点，不追求完整还原
4. 固定侧栏在窄屏可折叠，但主路径入口不能丢失

---

## 11. 可访问性与可用性底线

1. 文本与背景保持足够对比度
2. 交互控件有明确焦点态
3. 颜色不是唯一信息通道，状态需有文案或图标辅助
4. 表格字段可被快速扫描，避免密集难读

---

## 12. 当前实现对齐要求（落地约束）

为了与当前代码保持一致，后续迭代应遵循：

1. 优先复用现有样式令牌与类名（如 `ledger-*`、`pill`、`soft-table`）
2. 页面结构优先沿用 `AppShell` + 页面区块，不引入第二套壳层
3. 新组件优先放入现有 `components/*` 语义目录，避免重复 UI 基建
4. 图标、按钮、卡片、筛选条优先做“统一替换”，避免按页面零散修补

---

## 13. Do / Don’t（执行清单）

## 13.1 Do

1. 先搭信息层级，再上视觉细节
2. 先统一导航和图标语义，再做页面微调
3. 每次改版都验证“结论-趋势-明细-解释”链路是否完整
4. 让每个关键数字都能被解释和追溯

## 13.2 Don’t

1. 不要使用高对比硬边线把页面切成表格块
2. 不要在同一屏放多个视觉主焦点
3. 不要把公式直接甩给用户而没有解释步骤
4. 不要为对齐视觉稿而破坏当前真实数据链路与交互逻辑

---

## 14. 评审标准（用于后续验收）

每个页面评审时，至少回答以下问题：

1. 用户是否能在 5 秒内找到核心结论？
2. 关键数据是否有趋势与明细支撑？
3. 指标口径是否可解释且无歧义？
4. 与其他页面相比，交互和视觉语义是否一致？
5. 是否保持 Stitch 风格，同时兼容当前工程实现？
