# 收息佬前端完善计划（基于 PRD/SDD）

## 1. 目标

本文档用于把当前前端工作从“页面已有雏形”推进到“可按阶段落地并可验收”，并明确以下四件事：

1. PRD 功能需求与前端页面/模块的映射关系
2. 阶段执行依赖与前后顺序约束
3. 每个阶段可操作、可复验的验收标准
4. 首轮执行边界（本轮做什么、不做什么）

本文档只描述计划，不代表自动开始执行。

## 2. 依据与当前现状

### 2.1 依据文档

1. `docs/PRD.md`：功能范围、用户流程、MVP 验收标准
2. `docs/SDD.md`：页面映射、IPC 能力、状态管理、降级与错误处理

### 2.2 已完成

1. Electron + React + IPC 主链路已打通
2. 目录结构已完成第一轮简化
3. 概览页、个股详情页、自选页已有第一版视觉落地
4. 后端已接入真实数据源，不再依赖 mock

### 2.3 主要缺口

1. 图标语义和导航可识别性未系统统一
2. 核心页面跳转偏静态，分析路径未形成闭环
3. 详情页、对比页、回测页完成度不均衡
4. 空态/错误态/无数据态未形成统一组件口径
5. 缺少按 PRD/SDD 对齐的阶段验收口径

## 3. 需求映射（PRD/SDD -> 前端计划）

| PRD 必做能力 | SDD 页面/模块 | 前端落地页面 | 对应阶段 |
| --- | --- | --- | --- |
| 股票搜索与选择 | `stock-search`、`dashboard` | 概览页/搜索入口 | Phase 2, Phase 6 |
| 自选股管理 | `watchlist`、`watchlist:list/add/remove` | 自选页 | Phase 2, Phase 4 |
| 自然年股息率计算 + 年份筛选 | `stock-detail`、`calculation:historical-yield` | 个股详情页 | Phase 3 |
| 未来股息率估算（基准/保守） | `stock-detail`、`calculation:estimate-future-yield` | 个股详情页估算区 | Phase 3 |
| 多股票比较 | `comparison`、`stock:compare` | 对比页 | Phase 4 |
| 股息复投回测 | `backtest`、`calculation:run-dividend-reinvestment-backtest` | 回测页 | Phase 5 |
| 地图可视化（V1/增强） | `yield-map`、`visualization:yield-map` | 地图页 | Phase 4（预留入口）/后续专项 |

说明：

1. 当前计划优先覆盖 PRD 的 MVP 主链路（搜索 -> 详情 -> 自选/对比 -> 回测）。
2. 地图能力已纳入映射，但不作为首轮执行硬门槛。

## 4. 设计与实现原则

1. 先主路径闭环，再补外围页面
2. 先交互一致性，再做视觉精修
3. 计算口径以主进程服务返回为准，前端负责解释与呈现
4. 每阶段都可独立运行、独立验收、独立回滚
5. 未通过阶段验收，不进入下一阶段

## 5. 阶段依赖关系

### 5.1 业务依赖

1. `Phase 1 -> Phase 2`：导航语义统一后再做跳转闭环，避免路由行为反复返工
2. `Phase 2 -> Phase 3/4/5`：详情、对比、回测均依赖可传递的 symbol 与筛选上下文
3. `Phase 3 -> Phase 4`：对比页指标说明与口径复用详情页解释模块
4. `Phase 3 -> Phase 5`：回测页的“结果解释方式”复用详情页的口径说明框架
5. `Phase 4/5 -> Phase 6`：状态与组件收口放在功能稳定后执行

### 5.2 技术依赖（来自 SDD）

1. 路由：React Router 参数规范统一（`symbol`、`years`、`from`）
2. 状态：按领域 store（`useWatchlistStore`、`useStockDetailStore`、`useComparisonStore`、`useBacktestStore`）
3. IPC：仅使用 preload 白名单 API，不在页面直连 Node 能力
4. 异常降级：数据缺失/接口失败按“局部可用、全局可解释”策略展示

## 6. 分阶段计划（含验收口径）

## Phase 1：导航与图标补齐

### 目标

建立统一视觉语义，降低页面识别与切换成本。

### 范围

1. 左侧导航图标补齐
2. 顶部搜索区与操作图标补齐
3. 首页指标卡、快捷工具、自选状态标签图标补齐
4. 详情页、对比页、回测页关键卡片图标补齐

### 产出

1. 图标映射表（页面区域 -> 图标语义 -> 使用组件）
2. 图标规范（尺寸、颜色、激活态、禁用态）
3. 一级导航与关键操作语义统一

### 验收标准

1. 一级导航项均有图标且无重复语义冲突
2. 关键操作入口（搜索、加入自选、对比、回测）图标完整
3. 随机抽查三页无“同类操作不同图标”现象

## Phase 2：页面跳转闭环

### 目标

把静态页面连接为真实可用的分析路径。

### 范围

1. 概览页跳转至详情/自选/回测
2. 自选页行点击进入详情
3. 自选页批量选择进入对比页
4. 对比页点击股票进入详情
5. 详情页进入回测页
6. 保留 `symbol` 和关键筛选条件作为路由参数或可恢复状态

### 产出

1. 路由参数与页面上下文约定
2. 列表/按钮/卡片统一跳转行为
3. 可复用导航工具方法（含回跳与来源标识）

### 验收标准

1. 按 PRD 深度流程可连续走通：自选 -> 对比 -> 详情 -> 回测
2. 刷新页面后核心上下文（至少 `symbol`）可恢复
3. 不依赖硬编码默认股票作为唯一入口

## Phase 3：个股详情页细化

### 目标

把详情页打造成首个“可交付页面”。

### 范围

1. 年度股息率图完善（自然年口径明确）
2. 分红历史表优化（字段说明、缺失提示）
3. 未来股息率估算区“基准/保守”口径展示
4. 年份与区间切换交互完善
5. 空数据与部分缺失数据提示优化

### 产出

1. 历史股息率、估算股息率、解释链路三段式布局
2. 指标解释入口（口径、公式、字段来源）
3. 缺失字段的可解释降级提示

### 验收标准

1. 用户可区分“历史值”和“估算值”，且能看到计算链路
2. 自然年归属规则和估算方式（基准/保守）可见
3. 缺字段时不报错崩溃，提示明确缺失项

## Phase 4：自选页与对比页完善

### 目标

完善“筛选观察 + 多股分析”双页面协同能力。

### 范围

1. 自选页选择、批量操作、排序、过滤
2. 对比页指标排序、高亮、列说明
3. 自选与对比页共享表格交互能力
4. 补齐空态、无结果态、部分数据缺失态
5. 地图页入口预留（不展开完整地图能力）

### 产出

1. 可长期使用的自选工作台
2. 有分析意义的多股对比页
3. 表格交互与状态表达统一

### 验收标准

1. 可从自选批量进入对比并返回，不丢选中上下文
2. 对比页关键指标可排序且高亮逻辑正确
3. 空结果、缺字段、加载失败三类状态有区分提示

## Phase 5：回测页完善

### 目标

实现“结果 + 过程 + 假设”可读的回测页面。

### 范围

1. 买入日期选择
2. 回测触发与重新计算
3. 结果摘要卡优化
4. 回测流水可读性提升
5. 数据覆盖不足时提示
6. 回测假设说明固定展示

### 产出

1. 可交互回测页
2. 可追溯过程区（分红到账、复投、期末估值）
3. 明确的失败与降级提示

### 验收标准

1. 用户可修改条件并看到结果更新
2. 失败时返回可理解错误信息，不输出误导性结果
3. 页面可看到关键假设（买入价、复投时点、年化口径）

## Phase 6：统一状态与基础组件收口

### 目标

减少重复实现，控制后续迭代成本。

### 范围

1. 统一 `loading/empty/error/no-data` 组件
2. 统一图标按钮、筛选条、表格卡片、指标卡片
3. 收敛重复样式类和重复页面结构
4. 清理死代码与废弃展示组件

### 产出

1. 稳定可复用组件基座
2. 统一状态表达规范
3. 更低的新页面接入成本

### 验收标准

1. 跨页面状态组件表现一致
2. 重复 UI 片段明显下降（按组件复用替代页面拷贝）
3. 新页面接入基础态无需重复造轮子

## 7. 首轮执行边界（Round 1）

### 7.1 本轮纳入（In Scope）

1. `Phase 1` 与 `Phase 2` 全量完成
2. `Phase 3` 中与口径可解释直接相关的详情页核心区（历史图 + 估算区 + 说明区）
3. `Phase 4` 中自选到对比的核心闭环（批量进入、基础排序、高亮）
4. `Phase 5` 中回测最小可用闭环（选日期、触发、结果与过程展示）
5. `Phase 6` 中状态组件最小统一（至少覆盖详情/对比/回测三页）

### 7.2 本轮不纳入（Out of Scope）

1. 房价租售比、多资产扩展、账号与云同步
2. 地图页完整能力（Treemap/Heatmap 深化放到后续专项）
3. 前端新增计算口径或改写主进程公式
4. 高级导出能力（截图/图表导出）与复杂个性化配置
5. 非 Windows 平台适配

### 7.3 首轮完成定义（Round 1 DoD）

1. 覆盖 PRD MVP 主链路：搜索/详情/自选/对比/回测可连续走通
2. 关键页面均具备可解释状态：正常、空、缺失、失败
3. 评审可基于“路径录屏 + 参数恢复 + 异常演示”完成验收

## 8. 推荐执行顺序

建议按以下顺序推进：

1. `Phase 1` 图标补齐
2. `Phase 2` 页面跳转闭环
3. `Phase 3` 个股详情页细化
4. `Phase 4` 自选页与对比页完善
5. `Phase 5` 回测页完善
6. `Phase 6` 基础组件与状态统一

原因：

1. 先统一语义和路径，后做页面深加工，返工最小
2. 详情页是口径解释中心，先稳定后才能支撑对比与回测
3. 组件与状态收口放在后段，避免前期频繁改抽象

## 9. 阶段审核方式

每阶段开始前确认：

1. 本阶段范围是否与 PRD/SDD 映射一致
2. 前置依赖是否满足（路由/状态/IPC/数据）
3. 是否存在跨阶段需求挪入风险

每阶段完成后按以下证据验收：

1. 关键路径录屏（含跳转、返回、刷新恢复）
2. 页面状态截图（正常/空/缺失/失败）
3. 口径说明截图（详情估算、回测假设）
